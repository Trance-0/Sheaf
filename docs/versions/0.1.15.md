# Version 0.1.15

## Date range filter + per-edge alpha fade

This round replaces the single-value "days ago" `TimeScaleBar` with a
proper start/end date range picker and uses the selected `end` date as
the temporal anchor for an alpha fade on graph edges.

### New: `DateRangeFilter` component

- **[src/components/DateRangeFilter.tsx](src/components/DateRangeFilter.tsx)**
  is a new top-bar component with two native `<input type="date">`
  widgets styled with Tailwind. We use the browser-native calendar
  (`[color-scheme:light] dark:[color-scheme:dark]`) so the dependency
  footprint stays at zero — no date-picker library — and we inherit
  OS locale, keyboard nav, and accessibility for free.
- Exports a `DateRange` interface (`{ start: Date; end: Date }`) and a
  `defaultDateRange()` helper that returns `{ end: now, start: now - 1y }`.
- Clamps bidirectionally: if the user picks a `start` after the current
  `end`, `end` is bumped forward to match, and vice versa. This keeps the
  query window valid at every intermediate state during interaction.
- Renders a live span summary ("12 months", "2.0 years", "14 days") in
  the left badge.

### `/api/graph` accepts `?start=&end=`

- **[src/app/api/graph/route.ts](src/app/api/graph/route.ts)** — the
  `days=<N>` param is gone. The handler now reads `start` and `end` as
  ISO `YYYY-MM-DD` strings, parses them into `Date` objects, and filters
  events with `date: { gte: start, lte: end }`. If either param is
  omitted it falls back to `end = now` and `start = now - 1 year`, which
  matches `defaultDateRange()` on the client.
- Invalid dates return a 400 with `"Invalid start/end date. Expected
  ISO YYYY-MM-DD."` instead of silently returning an empty graph.

### Per-edge alpha fade in `GraphCanvas`

- **[src/components/GraphCanvas.tsx](src/components/GraphCanvas.tsx)** —
  `timeFilter: number` is replaced with `dateRange: DateRange`. The edge
  color is no longer a flat hex string — it's computed per-edge as
  `rgba(r, g, b, α)` where `α` is derived from the edge's *most recent*
  event date and its distance from `dateRange.end`:
  ```ts
  const FLOOR = 0.2;
  const span = end - start;
  const distance = max(0, end - latestEventOnEdge);
  const fraction = min(1, distance / span);
  const alpha = max(FLOOR, 1 - fraction * (1 - FLOOR));
  ```
  Edges whose latest event is close to `end` render near full opacity
  (~1.0). Edges whose events all sit near `start` fade toward the 0.2
  floor. A 0 floor would make old relationships vanish entirely, which
  is worse than a faint ghost — 0.2 is low enough to visibly recede but
  high enough to stay clickable.
- The latest-event anchor (rather than average) means an ongoing
  relationship with one fresh update still reads as "live" even if it
  also has old history, which matches how the user intuits recency.
- `IMPACT_RGB` holds the three base impact colors as RGB tuples so the
  rgba() composition path and the legacy hex path agree on the palette:
  positive `#10b981`, negative `#ef4444`, neutral `#6b7280`.

### Page wiring

- **[src/app/page.tsx](src/app/page.tsx)** — state changes from
  `useState<number>(9999)` to `useState<DateRange>(() =>
  defaultDateRange())`. The default window is now "last 1 year ending
  today" instead of "all time", which matches the user's explicit
  requirement. `TimeScaleBar` is replaced by `DateRangeFilter`; the
  `timeFilter` prop on `GraphCanvas` is replaced by `dateRange`.
- `GraphCanvas` serializes the range to ISO strings for both the fetch
  URL and the effect dependency array, so identity changes on the
  `Date` objects don't re-fire the effect on every parent render.

### Removed: `src/components/TimeScaleBar.tsx`

Deleted outright. The component had a single call site (`page.tsx`),
now replaced. Nothing else imports it.
