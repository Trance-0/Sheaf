# Version 0.1.20

## Frontend-triggered database migration page

v0.1.19 introduced a modal that caught the server's "backend is out of
sync" errors and showed the user a command to run in their terminal. For
a hosted Sheaf deploy where the user only supplies a database URL, that
is the wrong affordance — the user has no terminal, no repo clone, and
no way to run `npx tsx prisma/migrate_auto.ts`. v0.1.20 turns the same
modal into a full **in-browser migration page**: it fetches the list of
registered migration steps, gates the run behind an explicit "I've
backed up my database" acknowledgement, executes the bundled idempotent
raw-SQL upgrades against the user-supplied database URL, and renders a
per-step log + success/failure report inline.

### Architecture: shared migration list + structured report types

The CLI driver (`prisma/migrate_auto.ts`) and the new API route
(`/api/migrate`) need to run the exact same steps in the exact same
order, so the `MIGRATIONS` list is now lifted out of `migrate_auto.ts`
into a single canonical location:

- **[src/lib/server/migrations.ts](../../src/lib/server/migrations.ts)**
  owns the `MIGRATIONS` array and a new `runAllMigrations(prisma)`
  helper that executes every step under a capturing logger and returns
  a `MigrationReport`. Failures abort the loop at the first bad step —
  consistent with the CLI path and avoids cascading errors when later
  steps depend on earlier DDL.
- **[src/lib/migrationTypes.ts](../../src/lib/migrationTypes.ts)** (NEW)
  holds the pure TypeScript shapes — `MigrationStepMeta`,
  `MigrationStepResult`, `MigrationReport`. This file has zero runtime
  imports on purpose: both the server runner (which imports
  `PrismaClient`) and the client-side `BackendUpgradePrompt` (which
  runs in the browser) depend on it, and keeping PrismaClient out of
  the browser bundle requires the shared types to live in a
  dependency-free module.
- **[prisma/migrate_auto.ts](../../prisma/migrate_auto.ts)** now imports
  `MIGRATIONS` from `src/lib/server/migrations` instead of defining its
  own copy, and bumps the driver banner to v0.1.20. The CLI still owns
  the interactive `type yes to continue` prompt and the 5-second
  countdown; those are safety rails specific to the terminal workflow
  and don't belong in the shared library.

The underlying per-migration scripts
(**[prisma/migrate_snapshot_financials.ts](../../prisma/migrate_snapshot_financials.ts)**,
**[prisma/migrate_event_category.ts](../../prisma/migrate_event_category.ts)**)
grew an optional `logger: MigrationLogger` parameter that defaults to
`console`. The `MigrationLogger` type (`{ log: (message: string) =>
void }`) is declared inline in each file rather than imported from
`src/` so the prisma/ scripts remain importable from either direction —
the CLI path (argv guard) and the Next.js server path both work
untouched. Every `console.log(...)` call inside the migration bodies
was mechanically rewritten to `logger.log(...)`; the API route passes
in a logger whose `log` implementation pushes into a captured `string[]`
so each step's output can be returned to the browser.

### Layer 1 — `POST /api/migrate`

**[src/app/api/migrate/route.ts](../../src/app/api/migrate/route.ts)**
(NEW) wires the shared runner to an HTTP endpoint:

- **`GET /api/migrate`** — dry-run listing. Returns
  `{ migrations: listMigrations(), note }`. No database connection, no
  header required. Used by the migration page to populate "here's what
  will run" before the user confirms.
- **`POST /api/migrate`** — runs the migrations. Requires:
  1. `x-sheaf-database-url` header (same header every other Prisma-
     touching route already uses — no default database, no shared
     state).
  2. `{ confirmed: true }` in the JSON body. The frontend sets this
     flag only after the user ticks the "I've backed up my database"
     checkbox, matching the CLI's `type yes` prompt semantics.

`confirmed !== true` returns 400. A missing or malformed
`x-sheaf-database-url` surfaces via a `try/catch` around
`createPrismaFromRequest(req)` and also returns 400 — missing header is
a client error, not an upgrade-required condition, so routing it
through `backendUpgradeResponse` would be wrong.

The route returns **HTTP 200** even when individual migration steps
fail. Top-line HTTP status conveys "did the request reach the runner
and get a structured answer"; per-step failures are reported in the
body via the `MigrationReport.status` field. This is deliberate:
returning 5xx on a partial failure would cause the top-level `apiFetch`
wrapper (which listens for `BACKEND_UPGRADE_REQUIRED` on 5xx) to
dispatch another upgrade event and pop a second prompt on top of the
migration page the user is already looking at.

The top-level `catch` _does_ still route through `backendUpgradeResponse`,
but only for the narrow case where Prisma itself fails to initialize
before any migration step runs — that class of error (`prisma-client-
stale`) means the deployed server's generated client is broken, which
is a server deploy issue rather than a database issue and should
surface the "contact the maintainer" guidance rather than the
"run migrations" flow.

Prisma client lifecycle is owned by the route: `let prisma: PrismaClient
| undefined` up top, `prisma = createPrismaFromRequest(req)` inside the
try, and `await prisma?.$disconnect()` in the `finally`. The runner
does not touch lifecycle — that's the caller's responsibility.

### Layer 2 — `BackendUpgradePrompt` as a migration page

**[src/components/BackendUpgradePrompt.tsx](../../src/components/BackendUpgradePrompt.tsx)**
is now conditional on `info.reason`:

- **`schema-drift`** → full migration page (plan preview + backup
  acknowledgement + run button + inline report).
- **`prisma-client-stale`** → informational fallback explaining that
  this is a server-deploy issue rather than a database issue, with the
  same terminal hint the v0.1.19 modal surfaced.
- **anything else** → fallback shape (same as `prisma-client-stale`).

The schema-drift path walks through three phases, rendered
conditionally in a single modal that grows from `max-w-md` to
`max-w-2xl w-full max-h-[90vh]` with its own scroll container:

1. **Plan preview.** When `info.reason === "schema-drift"` first fires,
   a `useEffect` fetches `GET /api/migrate` (raw `fetch`, not
   `apiFetch` — the dry-run list shouldn't route through the upgrade-
   event wrapper) and renders the returned steps as a numbered list
   with `step.id` + `step.description`. Loading and error states are
   both handled inline.

2. **Backup acknowledgement.** An amber callout listing the three
   snapshot options from the CLI prompt (Neon branch, `pg_dump`, other)
   + a checkbox the user must tick. The checkbox controls a
   `backupAcknowledged` piece of state that the Run button reads. The
   whole section is hidden once the migration has finished so the
   report doesn't compete for vertical space with acknowledgement UI
   that's no longer relevant.

3. **Migration report.** After `runState` transitions to `"done"`, a
   new `MigrationReportView` renders the overall success/failure banner
   (total steps ran out of total registered, total elapsed seconds),
   then one collapsible `MigrationStepReport` per step. Each step shows
   its id + duration in the header; clicking expands to show the
   description, a dark-themed terminal-style log block (populated from
   the `logs[]` array the server captured), and — if the step failed —
   a red error block with the raw message. Failed steps start expanded
   so the user doesn't have to hunt for what broke.

The **Run Migration** button is disabled until all three conditions are
true:

```ts
!backupAcknowledged ||
!hasDatabaseUrl(settings) ||
runState === "running" ||
plan === null
```

`hasDatabaseUrl(settings)` reads from the same `useAppSettings` store
that Settings uses, so the modal picks up whatever URL the user already
configured locally without needing to re-prompt. If no URL is set, an
inline red line explains what's missing.

The run path uses raw `fetch`, deliberately bypassing `apiFetch`:

```ts
const res = await fetch("/api/migrate", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    ...buildDatabaseHeaders(settings),
  },
  body: JSON.stringify({ confirmed: true }),
});
```

If `apiFetch` were used and the migration somehow returned 5xx from the
top-level catch, the upgrade event would re-dispatch mid-run and latch
into a recursive "this prompt spawns this prompt" state. The migration
page owns its own error surface (`runError` state + a red banner in the
report section), so the detour is unwanted.

Footer actions are state-dependent:

- **idle / running** → Dismiss + Run Migration
- **done, success** → Close + Reload Sheaf (green emerald button —
  success state deserves a success color)
- **done, failed** → Dismiss + Retry (amber, since re-running is safe:
  every step is idempotent by construction)
- **prisma-client-stale fallback** → Dismiss + Reload after fix

### Why idempotency matters for the retry button

All three shipped migration steps use `ADD COLUMN IF NOT EXISTS`,
`CREATE UNIQUE INDEX IF NOT EXISTS`, or `UPDATE ... WHERE col IS NULL`.
A partial failure that aborts after step 1 leaves the database in a
state where re-running the full list is a no-op for step 1 and a fresh
attempt for step 2 — exactly what the Retry button assumes. If we ever
ship a non-idempotent migration, that contract will need a gate
(probably a per-step "has this been applied" marker in a
`_sheaf_migrations` table), but nothing currently shipped requires it.
