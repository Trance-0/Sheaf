# Version 0.1.16

## Settings import hardening + JobsConfig schema alignment

This round brings the in-app settings model into alignment with the
hand-written `sheaf-settings-2026-04-11.json` file and teaches
`importSettingsJson` to report mismatches instead of silently dropping
unknown fields.

### Schema: `JobsConfig` gets four renamed / new fields

**[src/lib/useAppSettings.ts](src/lib/useAppSettings.ts)**:

- `userResumeURL` → `resumeURL`
- `userJobKeywords` → `jobKeywords`
- **new:** `locationKeywords: string[]`
- **new:** `skillsKeywords: string[]`
- `userLevelOfExpertise` union now includes `"intern/entry"` (new
  default), alongside the legacy `"intern" | "junior" | "mid" | "senior"
  | "staff" | "principal"` values. Old settings files still round-trip.

`SETTINGS_VERSION` bumped `0.1.13` → `0.1.16`. The storage key
(`sheaf-settings-v2`) is unchanged — `normalizeSettings` gracefully
migrates older persisted blobs on hydrate.

### Import hardening: trailing commas + structured issue reporting

`importSettingsJson` used to swallow unknown fields and reject
trailing-comma JSON outright. Now it:

1. **Strips trailing commas** before `JSON.parse` via a character-wise
   walker (`stripTrailingCommas`) that respects string literals and
   escape sequences, so a `",}"` inside a string value is preserved
   while `, }` or `,\n}` between fields is dropped. This is not a full
   JSON5 parser — a real `,}` byte sequence inside a JSON string would
   be corrupted — but it's correct for every settings file shape Sheaf
   uses in practice, and avoids pulling in a JSON5 dependency.
2. **Collects structured issues** against an `EXPECTED_SHAPE`
   descriptor via a new `collectSettingsIssues(value, shape)` helper
   that walks the input and returns:
   - `missing` — field paths in the shape but absent from the input
     (e.g. `jobsConfig.skillsKeywords`)
   - `unknown` — field paths in the input but not in the shape (e.g. a
     stale `jobsConfig.userResumeURL` from a pre-0.1.16 export, or a
     typoed `extraTopLevel`)
   - `typeMismatch` — field paths present on both sides but with the
     wrong type (e.g. `jobsConfig.enabled: "yes"` instead of a boolean)
3. **Returns a structured outcome** (`ImportOutcome`) that carries an
   optional `issues: SettingsIssues` block alongside `ok: true`. The
   normalizer still runs and applies the imported settings, so the user
   gets the best-effort result *plus* a warning list — no more silent
   drops.

The `SettingsIssues` and `ImportOutcome` types are exported so the
panel can format them inline.

### SettingsPanel UI: new fields + warning panel

**[src/components/SettingsPanel.tsx](src/components/SettingsPanel.tsx)**:

- Jobs Config section now shows **four** textareas: Job Keywords,
  Location Keywords (new), Skills Keywords (new), plus the Resume URL
  input — bound to the renamed `resumeURL` / `jobKeywords` fields.
- Level-of-expertise dropdown adds `"intern/entry"` at the top.
- Import-result area now renders a dedicated amber-bordered box below
  the Import/Export row when `issues` is non-empty, grouped as
  "Type mismatches", "Unknown fields (ignored)", and "Missing fields
  (defaulted)". This is the mismatch-visibility surface the task asked
  for: when a user imports a file with renamed or removed fields, they
  see exactly what was dropped or defaulted.
- Import button toast summarizes counts:
  `"Imported with warnings: 4 missing, 3 unknown, 1 type mismatch."`.

**[sheaf-settings-2026-04-11.json](sheaf-settings-2026-04-11.json)** —

inside `researchConfig`. Strict `JSON.parse` now accepts the file
directly. (`stripTrailingCommas` would have rescued it anyway, but the

### Verification

- `npx tsc --noEmit` — clean.
- Sanity test script against `sheaf-settings-2026-04-11.json`:
  - Strict `JSON.parse` on the fixed file: **OK**.
  - Re-injected trailing comma: strict parse fails as expected;
    `stripTrailingCommas` then parse: **OK**.
  - `collectSettingsIssues(parsed, EXPECTED_SHAPE)`: `missing=[]`,
    `unknown=[]`, `typeMismatch=[]` — file matches the new shape
    exactly.
  - A synthetic broken file with renamed + wrong-type fields correctly
    reports `jobsConfig.userResumeURL`/`userJobKeywords` as
    **unknown**, `resumeURL`/`locationKeywords`/`jobKeywords`/
    `skillsKeywords` as **missing**, and `enabled: "yes"` as a
    **type mismatch**.
