### Task B — `migrate_auto.ts` one-shot migration driver

**[prisma/migrate_auto.ts](../../prisma/migrate_auto.ts)** is a new
top-level upgrade script that composes every idempotent raw-SQL
migration shipped to date under a single Prisma client, a single
confirmation prompt, and a single 5-second countdown. It's intended as
the "I cloned this onto a new box, point me at my old Neon and bring it
up to the current schema" button.

#### Refactor of the existing migration scripts

**[prisma/migrate_snapshot_financials.ts](../../prisma/migrate_snapshot_financials.ts)**
and
**[prisma/migrate_event_category.ts](../../prisma/migrate_event_category.ts)**
were refactored to export their migration body as a named function
(`runSnapshotFinancialsMigration`, `runEventCategoryMigration`) that
accepts an already-constructed `PrismaClient`. Each script keeps its
standalone CLI entrypoint, gated by a `process.argv[1]` regex check so
the entrypoint only fires when the file is invoked directly via `tsx`,
not when it's imported by `migrate_auto.ts`.

This way the two scripts remain individually runnable — the
previously-documented `npx tsx prisma/migrate_event_category.ts`
invocation is unchanged — and the auto-driver composes them without
spawning child processes or opening redundant Prisma connections.

#### Safety rails

The driver has two independent safety rails, both can be bypassed
only with explicit flags:

1. **Interactive backup confirmation.** Prompts with:

   ```
   ================================================================
     WARNING: this will run schema migrations against the database
     pointed to by DATABASE_URL (see prisma.config.ts / .env).

     Please BACK UP your database before proceeding.
     - Neon: create a branch from the current state.
     - Self-hosted Postgres: `pg_dump` to a safe location.
     - Any other: do whatever your normal snapshot process is.

     All migrations shipped here are idempotent (ALTER IF NOT EXISTS,
     UPDATE WHERE col IS NULL, CREATE UNIQUE INDEX IF NOT EXISTS) so
     running against an already-migrated DB is a no-op, but we still
     want a rollback option if something unexpected happens.
   ================================================================

   Type "yes" to continue, anything else to abort:
   ```

   Requires the literal response `yes` or `y` (case-insensitive) — we
   intentionally do NOT accept Enter-as-default because this prompt
   gates DDL against a production database. Bypass with `--yes` / `-y`
   for CI or follow-up runs.

2. **5-second countdown** after the prompt. Prints
   `Starting in Ns... (Ctrl+C to abort)` once per second on a single
   line (uses `\r`), giving the operator one last chance to kill the
   process. Bypass with `--skip-countdown`, which requires `--yes`
   (the countdown is the *last* safety rail, so you can't skip it
   without first bypassing the prompt).

#### CLI flags

| Flag | Meaning |
|---|---|
| `--yes` / `-y` | Skip the interactive "type yes" prompt. |
| `--skip-countdown` | Skip the 5-second countdown. Requires `--yes`. |
| `--dry-run` | Print what would run and exit. No DB connection. |
| `--help` / `-h` | Print usage and exit. |

#### Migration list

The driver currently registers two migration steps, in dependency
order:

1. **`0.1.13-snapshot-financials`** — adds
   `marketCapUsd`, `employeeCount`, `freeCashFlow`, `sourceName`,
   `sourceUrl` columns to `EntitySnapshot` (via
   `ADD COLUMN IF NOT EXISTS`) and creates a
   `EntitySnapshot_entityId_date_key` unique index on
   `(entityId, date)` via `CREATE UNIQUE INDEX IF NOT EXISTS`.
2. **`0.1.11-event-category`** — backfills `Event.category` as `'job'`
   for single-agency events, `'news'` for multi-entity or fallback
   events. Only touches rows where `category IS NULL`.

Adding a new migration to the driver is one new entry in the
`MIGRATIONS` array; the contract is
`(prisma: PrismaClient) => Promise<void>`.

### Task C — Level-of-Expertise dropdown + form field theming

The `<select>` for Level of Expertise and every other `<input>` /
`<textarea>` in the Jobs Config and Research Config columns of
**[src/components/SettingsPanel.tsx](../../src/components/SettingsPanel.tsx)**
were missing three theme-critical properties:

1. **Explicit text color.** Native form elements default to browser
   black, so in dark mode the text rendered as near-invisible
   black-on-dark. Fixed with `text-gray-800 dark:text-gray-100`.
2. **`color-scheme` hint.** Without this the native `<select>` arrow
   and `<input type="date">` calendar popover always render in light
   mode. Fixed with
   `[color-scheme:light] dark:[color-scheme:dark]`.
3. **`<option>` element background.** `<option>` does not inherit its
   parent `<select>`'s background — most browsers pop the dropdown as
   white-on-white in dark mode. Fixed by giving each option an explicit
   `bg-white dark:bg-slate-800` class.

All eight fields (resume URL, job/location/skills keyword textareas,
expertise select, primary entities, news sources, refresh period) now
route through two shared constants:

```tsx
const FORM_FIELD_CLASS = "mt-1 w-full px-3 py-2 rounded-lg bg-white/70 dark:bg-white/5 border border-slate-200 dark:border-white/10 text-sm text-gray-800 dark:text-gray-100 placeholder-gray-400 dark:placeholder-gray-500 outline-none focus:border-blue-500 transition-colors [color-scheme:light] dark:[color-scheme:dark]";
const OPTION_CLASS = "bg-white dark:bg-slate-800 text-gray-800 dark:text-gray-100";
```

which also picks up a focus ring and better placeholder contrast as a
side effect.

### Task C (cont.) — Date range filter polish

**[src/components/DateRangeFilter.tsx](../../src/components/DateRangeFilter.tsx)**
was flagged as "too plain" — two bare browser date inputs side by side
with a single label. This round:

- **Quick-range chips** in a secondary row: `1W`, `1M`, `3M`, `6M`,
  `1Y`, `3Y` (3Y caps out at the project's stated "past 3 years" data
  window). Each chip preserves the current `end` anchor and only shifts
  `start` backward — if you have end pinned to a specific date, chip
  navigation doesn't lose that anchor.
- **Active preset highlighting** via `activePresetLabel(range)`, which
  matches the current window width to its closest preset within a
  1-day tolerance (handles DST / leap-year / 30-vs-31-day drift). The
  active chip renders as a solid blue pill; the others are ghost chips
  that light up blue on hover.
- **Calendar icon overlay** inside each date input (absolute-positioned
  `<Calendar size={13}>` to the left of the text cursor), with a
  `group-focus-within:text-blue-500` transition so the icon tints
  active on focus.
- **Focus ring** via `focus:shadow-[0_0_0_3px_rgba(59,130,246,0.15)]`
  and a subtle hover border bump, so the inputs no longer look visually
  identical to a flat label.
- **Header icon chip** — the `Calendar` lucide icon is now inside a
  rounded blue-tinted chip (`bg-blue-500/10 border-blue-500/30`)
  instead of a bare monochrome stroke.

The span label (`"1 year"`, `"3 months"`, etc.) is unchanged. Native
`<input type="date">` is still the underlying control — no picker
library pulled in — we just dressed it up.

### Task E — GitHub repo + star-if-you-like footer

A small footer row now sits immediately after the Backup & Restore
section in `SettingsPanel.tsx`:

- **Left side:** `Trance-0/Sheaf` link with an inline GitHub mark
  (a minimal SVG — lucide dropped its branded `Github` icon for
  trademark reasons, so we inline the mark rather than pulling another
  icon dep). Clicking it opens the repo in a new tab.
- **Right side:** a "Star if you like it" pill with a `Star` lucide
  icon. Same target URL. The pill has an amber hover state
  (`hover:bg-amber-500/10 hover:border-amber-500/40
  hover:text-amber-700 dark:hover:text-amber-300`) to nudge the affordance.

Both links carry `target="_blank" rel="noopener noreferrer"` and sit
at `text-xs` so the footer stays unobtrusive — the panel is still
primarily about configuration, not marketing.
