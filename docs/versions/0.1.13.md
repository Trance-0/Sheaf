# Version 0.1.13

## Local JSON settings become the source of truth

This round removes the old backend-driven settings write flow and moves
Sheaf to a fully local JSON-backed configuration model.

### What changed

- **`src/lib/useAppSettings.ts`** now owns the full 0.1.13 settings schema:
  - `theme`
  - `nodeSizeFactor`
  - `edgeSizeFactor`
  - `databaseUrl`
  - `jobsConfig`
  - `researchConfig`
- **`src/components/SettingsPanel.tsx`** was expanded into a real editor for
  the full schema and now saves locally instead of posting to `/api/settings`.
- **`src/lib/server/prismaFromRequest.ts`** is new and builds a Prisma client
  from the `x-sheaf-database-url` request header.
- **`src/app/api/{graph,node,edge,jobs}/route.ts`** now read the DB URL from
  the request instead of relying on a deployment-time `DATABASE_URL` env var.
- **`src/components/GraphCanvas.tsx`** now sends the local DB URL header on
  graph loads, respects `edgeSizeFactor`, and shows a clear empty/error state
  when settings are missing.
- **`src/components/SidePanel.tsx`** now uses the same local DB URL flow for
  node/edge reads and entity edits.
- **`src/app/api/settings/route.ts`** now returns `410 Gone` to make it clear
  the old server-side persistence path is obsolete.
- **`README.md`** was rewritten to describe the local-first config model and
  document the 0.1.13 JSON schema.
- **`src/app/layout.tsx`** metadata was updated to match the new positioning.

### Why

Previously the app still depended on a backend settings write step to make
Prisma usable in deployment. That meant the product was not truly local-first
because the user's DB URL had to be staged into server state.

In 0.1.13 the user's local settings JSON is the source of truth instead.
The backend still exists, but it now behaves as a thin per-request query layer
instead of a hidden settings store.

### Verification target

- settings import/export should round-trip the full JSON schema
- graph loads once a local `databaseUrl` is present
- side-panel reads and entity edits use the same local DB URL path
- no server `.env` rewrite is required to use the app
