# Version 0.1.19

## Auto-save settings panel and backend-upgrade prompt

Two-task polish pass from the 2026-04-11 follow-up: the Settings panel
no longer needs a Save button, and the "stale Prisma client" /
"schema drift" class of 500s is now caught server-side and turned into
a dedicated modal on the frontend instead of a raw error string.

### Task A тАФ Remove the "Save Local Configuration" button

The Settings panel used to require an explicit click on
**Save Local Configuration** to commit any edits. That button is gone
тАФ the panel now auto-saves on a 400ms debounce any time the user
touches a field.

#### Implementation

**[src/components/SettingsPanel.tsx](../../src/components/SettingsPanel.tsx)**
gains two new pieces of state:

1. An `isUserEditRef` тАФ a `useRef<boolean>` that tracks whether the
   latest `draft` mutation came from a user interaction in this panel
   or from the `useAppSettings` store subscription.
2. A `patchDraft(patch)` helper that flips `isUserEditRef.current =
   true` before applying the partial update.

Every `onChange` / `onClick` handler was rewritten from
`setDraft((prev) => ({ ...prev, field: value }))` to
`patchDraft({ field: value })` (eight input sites + two theme toggle
buttons + two size-factor grids).

Two effects now cooperate:

1. **Sync-from-store effect** (`[settings]`): re-seeds the draft from
   `settings` whenever the store emits, **unless** `isUserEditRef.current
   === true`. This is what picks up the hydrated localStorage state on
   first mount and what picks up a fresh import тАФ and it stays out of
   the way during active editing.
2. **Debounced auto-save effect** (`[draft]`): waits 400ms after the
   last draft mutation, then calls `updateSettings(...)` with the full
   normalized config, then clears `isUserEditRef` so the subsequent
   store emit doesn't cause the sync effect to reseed and re-trigger
   another save.

This pairing was the critical piece: without the ref gate, the two
effects would ping-pong (save fires тЖТ store emits тЖТ draft reseeded тЖТ
save fires again), eventually stabilizing but burning a render per
keystroke. The ref makes the sync effect a no-op immediately after a
user-originated commit.

The React "avoid setState in effect" lint rule was surfacing on the
sync effect, which is the legitimate
"synchronize local state with external store" case effects exist for.
The single `setDraft(buildDraft(settings))` call has a scoped
`eslint-disable-next-line react-hooks/set-state-in-effect` with a
comment explaining why.

#### New export on the settings store

**[src/lib/useAppSettings.ts](../../src/lib/useAppSettings.ts)** gains a
non-hook `getSettingsSnapshot(): AppSettings` export that returns the
internal `state` directly. `handleImportFile` in SettingsPanel uses it
to re-seed the draft from the freshly-imported snapshot without waiting
for the effect cycle (since the ref is cleared at the same time as the
import).

#### What was removed

- `Save` icon import from lucide
- `saveSettings` function body (folded into the auto-save effect)
- `status` useState + the success-message `<p>` under Backup & Restore
- The entire footer `<div>` that held the Save button

### Task B тАФ "Backend needs upgrade" prompt

The server was still returning
`@prisma/client did not initialize yet. Please run "prisma generate"
and try to import it again.` as a raw `{ error, 400 }` body any time
the generated Prisma client got out of sync with the schema. The
frontend surfaced that as a red error line in the graph canvas, with
no hint to the user that the fix was to run `prisma generate` and
restart the dev server.

v0.1.19 introduces a three-layer pipeline: server-side detection тЖТ
HTTP envelope тЖТ client-side global modal.

#### Layer 1 тАФ Shared detector:
**[src/lib/server/backendErrors.ts](../../src/lib/server/backendErrors.ts)**

```ts
isBackendUpgradeError(err): BackendUpgradeReason | null
backendUpgradeResponse(err): NextResponse | null
```

The detector matches two classes of failure:

1. **`prisma-client-stale`** тАФ Prisma's
   `@prisma/client did not initialize yet` message, plus any error
   whose text references `prisma.*generate`. This is what fires when
   `prisma generate` was never run, or when the dev server is holding
   the old query-engine DLL in memory (the Windows EPERM lock case
   that v0.1.18 tried to fix at the migration-driver level but
   couldn't cover the "forgot to restart the dev server" path).
2. **`schema-drift`** тАФ Postgres raw-SQL errors
   `column "..." does not exist` / `relation "..." does not exist`,
   plus the Prisma error codes `P2021` (table missing) and `P2022`
   (column missing). These fire when the generated client is ahead of
   the actual database and the user needs to run
   `npx tsx prisma/migrate_auto.ts`.

Raw Postgres messages AND Prisma codes are both matched because
`$executeRawUnsafe` paths pass through the underlying pg error
verbatim instead of being wrapped in a typed Prisma error class.

`backendUpgradeResponse` wraps the detector and returns a 503
`NextResponse` with body:

```json
{
  "error": "<the raw message>",
  "code": "BACKEND_UPGRADE_REQUIRED",
  "reason": "prisma-client-stale" | "schema-drift",
  "hint": "Stop the Next.js dev server, run `npx prisma generate`, then restart it."
}
```

The `hint` is case-specific тАФ schema drift gets the
`npx tsx prisma/migrate_auto.ts` instruction from v0.1.18 instead.

#### Layer 2 тАФ Wiring into every Prisma-touching route

- **[src/app/api/graph/route.ts](../../src/app/api/graph/route.ts)** catch block
- **[src/app/api/node/route.ts](../../src/app/api/node/route.ts)** тАФ both GET and PATCH handlers
- **[src/app/api/edge/route.ts](../../src/app/api/edge/route.ts)** catch block
- **[src/app/api/jobs/route.ts](../../src/app/api/jobs/route.ts)** catch block

Each catch block now runs:

```ts
const upgrade = backendUpgradeResponse(error);
if (upgrade) return upgrade;
// ...existing 400 path unchanged
```

Non-upgrade errors (validation, Prisma `RecordNotFound`, etc.) still
hit the original 400/404 paths.

`src/app/api/settings/route.ts` is untouched тАФ it returns 410 Gone
and doesn't touch Prisma.

#### Layer 3 тАФ Client-side wrapper:
**[src/lib/apiFetch.ts](../../src/lib/apiFetch.ts)**

A drop-in `fetch` replacement. On any 5xx response it clones the body,
checks for `code: "BACKEND_UPGRADE_REQUIRED"`, and dispatches a
`CustomEvent<BackendUpgradeInfo>` on the window with event name
`sheaf:backend-upgrade-required` (exported as
`BACKEND_UPGRADE_EVENT`).

Crucially `apiFetch` always returns the original Response unchanged, so
existing `if (!res.ok) throw` error-handling paths in every caller
still fire and still display whatever inline error they'd normally
display. The modal is additive, not replacement.

Call sites migrated off raw `fetch`:

- **[src/components/GraphCanvas.tsx](../../src/components/GraphCanvas.tsx)** тАФ `/api/graph` fetch
- **[src/components/SidePanel.tsx](../../src/components/SidePanel.tsx)** тАФ `/api/node` (GET), `/api/edge` (GET), `/api/node` (PATCH)

#### Layer 3 (cont.) тАФ Global prompt component:
**[src/components/BackendUpgradePrompt.tsx](../../src/components/BackendUpgradePrompt.tsx)**

Mounted once in
**[src/app/page.tsx](../../src/app/page.tsx)** (no props, no state in
the parent). Listens for `BACKEND_UPGRADE_EVENT` in a `useEffect` and
renders an amber-themed glassmorphic modal when it fires.

Design notes:

- **Latches on first event.** If a flurry of failing requests all
  dispatch upgrade events, the modal only opens once тАФ repeated events
  after the first are ignored until the user dismisses.
- **Amber, not red.** The underlying database and user data are fine;
  the dev server just needs a kick. Using error-red would overstate
  the severity.
- **Reason-specific title.**
  `prisma-client-stale` тЖТ "Prisma client out of sync".
  `schema-drift` тЖТ "Database schema is behind".
  Everything else тЖТ "Backend upgrade required".
- **Code block for the hint.** Terminal icon + monospace font so the
  user knows exactly what to run.
- **"Show raw error" details block.** Collapsed by default тАФ the user
  shouldn't have to look at Prisma's internals unless they're
  debugging why the detector matched.
- **Two actions.** "Dismiss" (close the modal, everything else keeps
  running) and "Reload after fix" (`window.location.reload()` тАФ what
  the user will want after they run the command in another terminal).
