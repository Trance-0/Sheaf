# Version 0.1.14

## Historical financial time-series on EntitySnapshot

This round extends `EntitySnapshot` from an unused placeholder model
into a real monthly time-series of financial data for every entity in
the graph, backed by an explicit source URL per data point.

It is scoped to the **data layer**. The UI changes that consume this
data (date range filter, event alpha fade by distance from query end,
settings-import hardening, intern/new-grad job seed) land in follow-up
rounds and are tracked in [docs/TASK.md](docs/TASK.md).

### Schema: EntitySnapshot gets financial columns + provenance

- **[prisma/schema.prisma](prisma/schema.prisma)** — `EntitySnapshot`
  picks up five new columns:
  - `marketCapUsd  Float?`
  - `employeeCount Int?`
  - `freeCashFlow  Float?`
  - `sourceName    String?` — short human label ("Yahoo Finance",
    "Apple 10-K FY2021", "Bloomberg")
  - `sourceUrl     String?` — direct link to the underlying source
  - plus `@@unique([entityId, date])` so the backfill can upsert
    idempotently on (entity, month).
- The legacy columns `netWorth`, `growth`, `statusText` are left in
  place for backward compatibility; new code should prefer the
  explicit columns.
- **[prisma/migrate_snapshot_financials.ts](prisma/migrate_snapshot_financials.ts)**
  runs the actual `ALTER TABLE` + `CREATE UNIQUE INDEX` via
  `$executeRawUnsafe` + `IF NOT EXISTS` so it's safe to re-run and
  independent of whatever generated prisma client is on disk. Ran
  clean: 5 columns added, 1 unique index created.

### Backfill: Yahoo Finance monthly + hand-curated annuals

- **[skills/financials_crawl/seed_historical_financials.ts](skills/financials_crawl/seed_historical_financials.ts)**
  is a two-pass backfill script:
  1. **Yahoo Finance pass.** Hits
     `query1.finance.yahoo.com/v8/finance/chart/<ticker>?interval=1mo&range=max`
     for every entity with a `stockTicker`, dedups by ticker (AWS and
     Amazon both map to `AMZN`; Google and Alphabet both map to
     `GOOGL`), and derives a stock-price-indexed market cap proxy via
     `close × (currentMarketCap / currentClose)`. This matches today's
     market cap exactly and fades backward proportionally to the stock
     price. It is **not** a strictly-correct historical market cap —
     shares outstanding actually changed over time — but it's
     sufficient for relative sizing in the graph, and the approximation
     is documented in the per-row `sourceName`.
  2. **Manual pass.** Reads
     [skills/financials_crawl/manual_snapshots.json](skills/financials_crawl/manual_snapshots.json),
     a hand-curated file of 74 annual employee / free cash flow points
     (from SEC 10-K filings) plus private-company valuation rounds
     (Anthropic, OpenAI, Figma, ByteDance — from Bloomberg / TechCrunch /
     Reuters / WSJ / The Information coverage). Every point carries a
     `sourceName` + `sourceUrl`.
- Idempotency: both passes go through a single `upsertSnapshot` helper
  that does `INSERT ... ON CONFLICT ("entityId", "date") DO UPDATE`
  with `COALESCE(EXCLUDED.x, existing.x)` for the numeric columns, so
  a Yahoo pass followed by a manual pass doesn't clobber market cap,
  and a manual annual employee point updates in place on the matching
  Yahoo month row.
- Raw-SQL approach: the script uses `prisma.$executeRaw` instead of the
  typed client because the generated `@prisma/client` on disk is
  frozen to the old `EntitySnapshot` shape (no `marketCapUsd` column,
  no composite unique), and `prisma generate` is blocked on Windows
  when the dev server holds the query engine DLL. Next time the dev
  server is restarted, `prisma generate` will pick up the new columns
  and the frontend can switch to typed reads.

### Backfill results

```
Yahoo pass: 4742 upserts across 17 distinct tickers, 0 failures
Manual pass: 74 upserted, 0 skipped for missing entity

Final EntitySnapshot totals:
  rows: 3981
  with marketCapUsd: 3973
  with employeeCount: 63
  with freeCashFlow: 52
```

The row count is smaller than Yahoo upsert count because 2 pairs of
entities share tickers (aws+amazon → AMZN, google+alphabet → GOOGL) —
the first upsert inserts, the second updates the same row in place.

### Not in scope (bumped to 0.1.15 – 0.1.17)

- **Date range filter bar + alpha fade** — `TimeScaleBar` still
  accepts a single "days ago" value. Swapping it for a start/end date
  range picker and wiring `/api/graph` to accept `?start=&end=`
  is a follow-up.
- **Settings import hardening** — `importSettingsJson` still silently
  drops unknown fields. The existing
  [sheaf-settings-2026-04-11.json](sheaf-settings-2026-04-11.json) file
  uses field names (`jobKeywords`, `skillsKeywords`, `locationKeywords`,
  `resumeURL`) that don't match the current `JobsConfig` type, and the
  file also has an invalid trailing comma that would make `JSON.parse`
  throw. Aligning the schema + reporting field-level mismatches is a
  follow-up.
- **Intern / new-grad job seed** — the `skills/job_crawl/update_jobs.ts`
  CLI has no notion of entry-level roles or skill keyword filtering.
  Building a filtered seed that uses the `skillsKeywords` from the
  active settings file is a follow-up.

All three are listed in [docs/TASK.md](docs/TASK.md).
